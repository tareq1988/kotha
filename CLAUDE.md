# CLAUDE.md

Guidance for working in this repo. Kotha is a minimal macOS menu-bar voice-dictation
app (SwiftUI, XcodeGen). See README.md for the feature overview.

## Build / run

```bash
./run.sh      # quick Debug build + launch from ./build
./build.sh    # Release build → re-sign → install to /Applications → launch
```

- **The project is generated by XcodeGen from `project.yml`.** After adding, removing,
  or moving source files you MUST run `xcodegen generate` before building.
  `build.sh` only regenerates when `project.yml` is newer than the `.xcodeproj`, so
  when in doubt run `xcodegen generate` explicitly.
- `Kotha.xcodeproj` is generated and git-ignored — never edit it by hand.
- SourceKit "Cannot find type X in scope" diagnostics while editing are usually
  stale per-file noise; trust a clean `build.sh` over them.
- Target macOS 14; the user runs macOS 26. CoreSimulator warnings in build output
  are unrelated noise — filter them.

## Commits — Conventional Commits (required)

Format: `type(scope): description`. A `commit-msg` hook enforces it
(`git config core.hooksPath .githooks`, already enabled). Types: feat, fix, perf,
refactor, docs, style, test, build, ci, chore, revert. See CONTRIBUTING.md.
Only commit or push when the user asks.

## Releasing

One command, fully documented in RELEASING.md:

```bash
./release.sh <version> [notes]      # e.g. ./release.sh 1.2
```

It bumps versions, builds, EdDSA-signs, updates `appcast.xml`, tags, pushes, and
creates the GitHub release. Notes auto-generate from conventional commits if omitted.
**Never hand-edit version numbers for a release — let the script bump them.**
`CURRENT_PROJECT_VERSION` (build number) must increase every release; Sparkle
compares it. Both version keys are wired to build settings via `$(…)` in
`project.yml` — keep them that way or bumps won't propagate to the built app.

## Project layout (`Sources/`)

`App/` (entry, Info.plist, entitlements, Assets) · `Core/` (AppState, AudioRecorder,
WAV) · `Transcription/` (model catalog, ModelManager/DownloadManager, STT engines) ·
`Cleanup/` (AI vocabulary cleanup) · `Data/` (JSONStore, Secret/History stores,
Pricing) · `System/` (hotkeys, paste, permissions, dock, updater) · `UI/` (windows,
views, HUD). Shared SwiftUI helpers live in `UI/AppUI.swift`.

## Conventions & gotchas

- **Persistence:** all app data is JSON files under `~/Library/Application Support/Kotha`
  via `JSONStore` (background writes). API keys live in `keys.json` (chmod 600) — NOT
  the Keychain (the Keychain re-prompts on every rebuild). Don't reintroduce Keychain.
- **Signing:** self-signed identity "Kotha Code Signing" (from `setup-signing.sh`);
  `build.sh` re-signs with it so Accessibility/Mic permissions persist across builds.
  Not notarized → first-launch Gatekeeper warning is expected.
- **Bundle id:** `co.tareq.kotha`.
- **Managers** (`ModelManager`, `CleanupManager`) subclass `DownloadManager` for shared
  download/progress/delete state. Downloadable-model rows share `DownloadControl`.
- **Menu-bar-only app** (LSUIElement); the Dock icon appears only while the main
  window is open (`DockPolicy`).
- Keep changes idiomatic to surrounding code; prefer DRY/KISS. The user cares about
  clean, well-organized code and premium-feeling UI.
